## Security Fundamentals

I spent the first four weeks on the CIA triad — confidentiality, integrity, availability. From there I dug into attack taxonomies: phishing, privilege escalation, denial of service, man-in-the-middle. For each one, I traced the kill chain — the sequence of steps an attacker takes from initial access to objective — and mapped where defenses should sit.

## Command Line and Linux

One exercise had me trace which users had accessed a sensitive directory over the past week using nothing but log files and `grep`. From there it was navigating Linux file structures, managing permissions with `chmod` and `chown`, writing basic Bash scripts to automate file auditing tasks. I was comfortable in the terminal already, but this was the first time I looked at every permission bit and asked "what could an attacker do if this were misconfigured?"

## Risk Assessment

The exercise that stuck with me was assessing a small business network where the biggest vulnerability turned out to be an unpatched printer with network access. Nobody thinks about the printer. That's the point. I worked through the full risk assessment framework — identifying assets, scoring threats by likelihood and impact, recommending controls — but the printer is what made the lesson land.

## Thinking Like an Attacker

When I write code, I think about what should happen. Security requires thinking about what shouldn't happen — and what someone with bad intentions would try. That shift showed up everywhere: reviewing firewall rules not for what they allow, but for what they fail to block; looking at a login form and immediately thinking about credential stuffing rather than UX.

Next month covers network security — packet analysis with Wireshark, hands-on labs, and the first real deep dive into how traffic moves (and gets intercepted) across a network.
