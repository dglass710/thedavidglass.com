## Security Fundamentals

The first four weeks covered the CIA triad — confidentiality, integrity, availability — and how every security decision maps back to those three concepts. We worked through attack taxonomies: phishing, privilege escalation, denial of service, man-in-the-middle. For each one, we walked through the kill chain and discussed where defenses should sit.

## Command Line and Linux

Most of the hands-on work lived in the terminal. We navigated Linux file structures, managed permissions with `chmod` and `chown`, and wrote basic Bash scripts to automate file auditing tasks. One exercise had us trace which users had accessed a sensitive directory over the past week using nothing but log files and `grep`. Coming from a software development background, I was comfortable in the terminal, but thinking about file permissions as an attack surface was a new lens.

## Risk Assessment

We also ran through risk assessment frameworks — identifying assets, scoring threats by likelihood and impact, and recommending controls. The exercise that stuck with me was assessing a small business network where the biggest vulnerability turned out to be an unpatched printer with network access.

## Takeaway

The hardest shift so far has been thinking like an attacker instead of a builder. When I write code, I think about what should happen. Security requires thinking about what shouldn't happen — and what someone with bad intentions would try.
