## Overview

I built an open-source SOAR lab on Shuffle to automate the triage layer between a SIEM and a human analyst. Two playbooks do the work. Playbook 1 takes a reported phishing email, extracts every URL, checks each against URLhaus, and opens a TheHive case with URL observables if any are malicious. Playbook 2 takes a simulated Wazuh alert containing a source IP, queries AbuseIPDB for a confidence score, and routes the alert down one of three branches — benign, suspicious, or malicious — creating a TheHive case at severity 2 (Medium) or severity 4 (Critical) depending on the score. The benign ones get logged and closed. The analyst's queue only sees what needed judgment.

Two Docker Compose stacks run side by side on the host: Shuffle as the orchestration engine, TheHive as the case-management platform. Shuffle reaches TheHive over `http://host.docker.internal:9000` — deliberately decoupled at the network layer, which matches how enterprise SOCs keep SOAR and their case tooling on separate subnets. Every pattern in the lab — parallel IOC enrichment, fan-in verdict aggregation, threshold-based branching, REST-based case creation — maps 1:1 onto Splunk SOAR, Cortex XSOAR, and QRadar SOAR.

## The Architecture

An alert enters Shuffle — in the lab via manual trigger, in production via webhook from the SIEM. The first node parses the payload and extracts the IOCs (URLs from the email body, or the source IP from the alert JSON). The next node iterates the IOC list in parallel against the threat-intel API; `.#` is Shuffle's list-iteration marker, the same concept as `phantom.collect()` in Splunk SOAR or a ForEach transformer in XSOAR. A Python node collapses the parallel responses into a single verdict. Edge conditions fire the matching branch, which issues two sequential POSTs to TheHive: the first creates the case, the second — iterated — attaches an observable per IOC. The analyst opens TheHive and sees a case with every IOC pre-attached.

## Playbook 1 — Phishing Triage

The input is a fake phishing email body with two test URLs chosen to exercise real patterns: `paypaI-secure.com` (homoglyph — capital-I masquerading as lowercase-l) and `malicious-login.example` (reserved TLD for documentation). A regex node extracts them with `https?://\S+`. The extracted list fans out to URLhaus, abuse.ch's free URL-reputation feed — one POST per URL in parallel. URLhaus returns `query_status: "ok"` when a URL is in its malware-URL database, `"no_results"` when it isn't. A Python aggregator reads the combined URLhaus output as a single raw string and substring-matches `"query_status": "ok"` to decide "malicious" or "clean." On malicious, Shuffle POSTs a TheHive case at severity 3 (High) with tags for the SOAR source, playbook, verdict, and intel source, then iterates a second POST to attach each URL as a `url`-type observable. On clean, the branch logs and closes.

## Playbook 2 — Suspicious IP Enrichment

The input is a simulated Wazuh alert JSON with a `source_ip` field set to `185.220.101.50` — a well-known Tor exit node, chosen because it gives a stable, high-confidence malicious signal from AbuseIPDB every time. A GET to `https://api.abuseipdb.com/api/v2/check?ipAddress=$alert.source_ip&maxAgeInDays=90` returns the reputation record. AbuseIPDB uses a custom `Key:` header for auth, not `Authorization: Bearer` — every API picks its own style, and guessing is the most common 401 trap in integration work. A Python node regex-extracts `abuseConfidenceScore` from the response and classifies it: ≥75 is malicious, 25–74 is suspicious, <25 is benign. The node emits structured JSON (`{"score": 89, "verdict": "malicious"}`) so downstream edges can read `$decision.verdict` cleanly instead of reparsing. Three edges leave the decision node, one per tier. Malicious opens a TheHive case at severity 4 (Critical) with the IP attached as an observable; suspicious opens one at severity 2 (Medium) and does the same; benign logs and closes.

On the Tor-exit-node test, AbuseIPDB returned score 89, `isTor: true`, 85 reports from 47 distinct users — routed malicious, case created, observable attached, `X-RateLimit-Remaining: 999` out of AbuseIPDB's daily 1,000 free lookups.

## Cases and Artifacts Live on Different Clocks

Both playbook branches end the same way — one POST to create a case, then one or more POSTs to attach observables. At first I wrote that off as a quirk of TheHive's REST layout. It isn't. A case and an observable are different kinds of objects, and the two-step pattern reflects the fact that they live on different clocks.

A case is an investigation with a lifecycle: it opens, an analyst works it, it closes. An observable — an artifact, an IOC — doesn't have that arc. The case that introduced it may close tomorrow, but the URL or IP sits in TheHive's database indefinitely, available for correlation against every future alert. When the next phishing case comes in two months later and one of its URLs matches an observable from an old case, the analyst sees the history. That's exactly the value of case-management tooling — not the current investigation by itself, but the accumulated record every prior investigation contributes to.

That's why a standard SOAR workflow does the two POSTs as distinct steps. The case creation step is bounded — it sets up the investigation. The observable step isn't final; more observables can be added by the analyst as they dig in, by later playbooks triggered off the same alert, or by cross-correlation from other cases. The workflow I wrote creates the case and attaches the IOCs the triage stage already knows about. Everything after that belongs to the analyst and the platform's history.

## The Bug That Taught Me How Shuffle Thinks

The aggregator node in Playbook 1 was supposed to fan two URLhaus responses into one verdict and fire the right branch. Instead it ran twice — once per URL. Each execution saw only one response, computed its own local verdict, and fired its own branch. The fan-in never happened.

The bug was in a reference that looked innocuous. My first draft of the aggregator was `$urlhaus.#.body.query_status` — drill into each iteration's response and grab the field. The trap is that `.#` in Shuffle isn't just "list-iteration syntax." It's a context modifier that propagates downstream. When a downstream node's variable reference contains `.#`, the node itself inherits iteration from upstream. The aggregator ran per-URL because I'd told it to, without realizing I had.

The fix looks odd until you know what it's avoiding:

```python
raw = """$urlhaus"""
print("malicious" if '"query_status": "ok"' in raw else "clean")
```

Reference `$urlhaus` without `.#` and Shuffle hands back the entire upstream output — all iterations, as a single raw string. The triple-quoted Python string handles the multi-line JSON without escaping. A substring check on the hit signature finds a malicious result anywhere in the blob. One node execution, one verdict, one branch fired. The code looks suspicious because it doesn't parse anything structured; it's written that way because structure was precisely what iteration inheritance was stripping.

Every SOAR platform has this trap in some form. Splunk SOAR's `phantom.collect2()` vs. per-artifact references, XSOAR's `dt` context scoping vs. loop scope, QRadar SOAR's datatable-row context vs. workflow context — the vocabulary differs, the hazard is identical. Nine other bugs from the build are written up in the repo's lessons-learned doc.

## Mapping to Enterprise SOAR

Shuffle is open-source and self-hostable, which is why the lab lives there. Every workflow concept the playbooks actually use — `.#` iteration, the decision node, the HTTP integrations, the case-creation POSTs — has a direct equivalent in the commercial platforms.

| Shuffle concept | Splunk SOAR (Phantom) | Cortex XSOAR (Demisto) | QRadar SOAR (Resilient) |
|---|---|---|---|
| Workflow | Playbook | Playbook | Workflow |
| Node | Action block | Task | Function / Script |
| `.#` list iteration | `phantom.collect()` | ForEach transformer | Loop over datatable |
| Edge condition | Decision block | Conditional task | Decision point |
| HTTP app | REST Data Source | Generic HTTP integration | REST function |

The vocabulary shifts across vendors but the primitives are the same: orchestration engines have converged on the same small set of building blocks.

## What I'd Change

I scoped a third playbook out of this build — a VirusTotal file-hash check with an auto-contain branch — to finish TheHive integration on the two I already had. Two fully-integrated playbooks with real case creation and observable attachment beat three half-wired ones. Playbook 3 is a straightforward port of the same patterns, and it's the next thing I'll add.

The bigger change is replacing manual execution with a webhook trigger. Right now I kick off each workflow by hand in the Shuffle UI; the production shape is a webhook endpoint that accepts SIEM alert JSON, which Shuffle deserializes into the same `$alert.*` fields the playbooks already read. Same playbooks, real trigger path. A Wazuh instance writing real detections into that webhook, rather than a hardcoded JSON payload, is the logical follow-on.

I'd annotate each playbook with MITRE ATT&CK technique IDs — T1566 (Phishing) for the phishing triage, T1110 (Brute Force) for the SSH-failed-login alert — so each case lands tagged with the tactic it's responding to. That metadata is how teams roll SOAR data into coverage heatmaps: which techniques the SOC has detected recently, which are going unaddressed.
